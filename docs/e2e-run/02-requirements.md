# Requirements — minimal torch-only DDIM-variant scheduler with configurable `prediction_type`

Stage: brainstorm (PM). Date: 2026-06-21 (round r3). Input: `docs/e2e-run/01-ideation.md` (recommendation =
Option 1, copy-adapt the real exemplar; no new abstraction). This spec answers **WHAT** to build; `plan`
answers HOW.

**Golden task:** "Add a minimal torch-only scheduler: a DDIM variant with a configurable `prediction_type`."

## Summary

Deliver a new, self-contained torch-only scheduler that is a copy-adaptation of the closest existing prior
art, `src/diffusers/schedulers/scheduling_ddim.py`, into a new file `scheduling_ddim_variant.py` exposing a
new public class `DDIMVariantScheduler` (with its paired output dataclass `DDIMVariantSchedulerOutput`). The
new scheduler keeps the exemplar's existing `prediction_type: Literal["epsilon", "sample", "v_prediction"]`
config field (`scheduling_ddim.py:204`) and its verbatim three-branch dispatch in `step`
(`scheduling_ddim.py:455-466`). No new base class, no shared abstraction, no mutation of `DDIMScheduler` —
the single-file policy prescribes copy-paste + `# Copied from` over abstraction (authority: `PHILOSOPHY.md` →
`docs/source/en/conceptual/philosophy.md:39-40,47,50` general single-file policy; `:101,103`
schedulers-specific: schedulers must stay self-contained, must not import from large utils files, and reuse
via `# Copied from`). The work is mechanical (new file + registration + tests); all failure modes are caught
by existing gates (`make quality`, `make fix-copies`).

This is a Lightweight, well-bounded option: one new component, one component type, no public-API change to
shipped code. The dialogue beat is degenerate (the option is fully specified and source-confirmed against the
live tree), so the spec proceeds straight to `plan`.
